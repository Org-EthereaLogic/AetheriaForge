## Root Cause of the Remaining Linker Failure

With the filter bug fixed, the downstream `gh api POST
orgs/${GITHUB_REPOSITORY_OWNER}/artifacts/metadata/storage-record` call
finally got to run for the first time, and immediately returned HTTP 422:

```
gh: Invalid request.
Invalid property /github_repository: Org-EthereaLogic/AetheriaForge
does not match `/^[A-Za-z0-9.\-_]+$/`. (HTTP 422)
```

The GitHub artifact-metadata storage-record endpoint requires
`github_repository` to be a bare repository identifier (no slash). The
Python script was passing the full `owner/repo` form from
`GITHUB_REPOSITORY`, which contains `/` and therefore fails the
`^[A-Za-z0-9.\-_]+$` regex on the API side.

This bug existed in every prior release, but it was hidden behind the
earlier filter bug: the Python script always `SystemExit`-ed on the
`.gitignore`/`.publish.attestation` comparison before it could reach the
bash loop that calls the API. Fixing the filter unmasked this second bug.

## Follow-Up Fix

Landing on `main` after this record is written:

- `.github/workflows/publish.yml` — change the payload assignment to
  `"github_repository": os.environ["GITHUB_REPOSITORY"].split("/", 1)[-1]`
  so only the repository name is submitted (`AetheriaForge` in this repo).
- `tests/test_packaging_metadata.py` — add a regression assertion that the
  shipped workflow strips the owner prefix, so the bug cannot silently

## No v0.1.5 Patch Release

PyPI will reject a re-upload of `0.1.4` (same version cannot be published
twice), and cutting a `0.1.5` just to re-trigger the workflow would be
release-history noise without any customer-visible change. The workflow fix
therefore lands on `main` only; it will first take effect on the next real
release cut from `main`.
